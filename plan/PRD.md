# flash-deal — PRD

## 1. 背景

秒杀(Flash Deal)是国内互联网后端面试**最高频考点**之一,因为它在一个小场景里压缩了大厂后端的核心能力:
- 高并发(瞬时流量)
- 一致性(不能超卖)
- 异步化(削峰填谷)
- 限流熔断
- 缓存策略
- 分库分表
- 幂等设计

本项目目标是做出一个**面试可讲、可压测、有数据**的秒杀系统。

## 2. 受众
- **第一受众**:大厂后端面试官
- **第二受众**:博客读者
- **自我目标**:把分布式系统知识从纸面落到代码

## 3. 范围

### In Scope
- 秒杀下单核心链路(限流 → 库存 → 异步落单 → 查询)
- Redis Lua 原子扣减
- Kafka 异步削峰
- MySQL 分库分表(订单表水平拆分)
- 限流(令牌桶 + 漏桶)
- 熔断(gobreaker)
- 幂等(token + 唯一索引)
- 全链路 trace + metrics
- 完整压测体系

### Out of Scope
- 完整电商前端(只做接口 + 简单管理页)
- 支付集成(用 mock)
- 物流 / 售后
- 推荐 / 搜索

## 4. 功能需求

### F1. 秒杀活动管理
- 创建活动(商品 / 库存 / 开始时间 / 结束时间 / 限购)
- 库存预热(活动开始前把库存写入 Redis)

### F2. 秒杀下单 `POST /v1/seckill`
请求体:
```json
{
  "activity_id": 1001,
  "user_id": 88888,
  "idempotency_token": "uuid-xxx"
}
```
处理:
1. 限流(用户级 5 rpm、活动级 5w qps、全局 10w qps)
2. 校验活动是否在售卖窗口
3. 校验幂等 token
4. Redis Lua 原子扣减库存(返回剩余库存或失败)
5. 投递 Kafka `seckill_orders`,返回排队 token
6. 同步返回 `{"status": "queued", "queue_token": "..."}`

### F3. 异步消费下单 `cmd/consumer`
- 从 Kafka 消费消息
- 写订单(分库分表路由)
- 失败重试(死信队列)

### F4. 查询订单状态 `GET /v1/order/by-token/:token`
- 用户轮询查询(或 SSE 推送,可选)

### F5. 防刷与风控(基础版)
- 同 IP 频率限制
- 同 user 频率限制
- 黑名单(配置)

### F6. 监控大盘
- 实时 QPS、成功率、延迟分布
- Redis / MySQL / Kafka 关键指标
- Grafana dashboard

## 5. 非功能需求

| 维度 | 目标 |
|------|------|
| 单机 QPS | ≥ 30,000(下单接口) |
| 集群 QPS | ≥ 100,000(2-3 实例 + LB) |
| P99 延迟 | ≤ 50ms |
| 库存正确性 | **绝对不超卖、不少卖** |
| 幂等 | 同 token 重复请求结果一致 |
| 可观测 | 全链路 trace 可见 |
| 可复现 | `make up` 一键起 |

## 6. 成功指标
- 压测报告 PDF/MD,带 Grafana 截图
- 库存正确性验证脚本(并发 10w 下库存最终值 = 初始值 - 订单数)
- 简历句:"独立设计 Go 秒杀系统,单机 QPS 30k+,P99 < 50ms,通过 Redis Lua + Kafka 实现零超卖"
- GitHub 50+ star 或博客 5000+ 阅读

## 7. 风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 本地机器扛不住压测 | 数据失真 | 开多机分布式压测,或 GCP 短租 |
| Kafka / MySQL 配置不当成瓶颈 | 假性能 | 提前 baseline 测,优化前后对比 |
| Lua 脚本 bug 导致超卖 | 项目失败 | 单元测试 + 并发测试覆盖 |
| 时间超期 | 简历来不及用 | MVP 先跑通,优化迭代式追加 |
